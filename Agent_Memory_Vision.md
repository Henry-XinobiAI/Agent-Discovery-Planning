# Agent Memory Vision

> 개인의 모든 활동·기억을 시간적·의미적 맥락으로 구조화한 **personal knowledge graph**로 만들고,
이를 외부 공개 knowledge graph(Wikidata 등)에 정렬한 **public anchor 네트워크**와 연결해
"어떤 주제든 누가 가장 잘 아는지"를 찾아 묻고 답하는 지식 시스템을 구축한다.
> 

핵심 통찰: **공개 지식은 정답을 직접 보관하는 저장소가 아니라, 그 주제를 가장 잘 아는 개인·단체로 향하는 anchor(연결점)의 집합이다.** 공개 지식이 담는 것은 기본적·핵심적 사실, 그리고 그 사실을 깊이 아는 곳으로 이어지는 링크다.

---

## 1. 무엇을 다루는가

**agent memory** — 이 프로젝트가 다루는 대상은 한 사용자의 **기억·경험·지식** 전체다.

핵심 원칙:

- **소유·관리**: 모든 것은 **사용자가 소유하고 관리한다.**
- **접근 통로**: 기억·경험·지식의 접근·조회·활용은 **항상 그 사용자의 agent를 거쳐서만** 가능하다.
- **공개 범위**: 개별 기억·내용 자체는 공개하지 않는다. 본인을 소개하거나, 전문성을 판별하거나, 상황에 맞는지 확인할 수 있는 정도의 **간략한 description 수준만** 공개한다.

agent memory는 우선 **conversations**를 다루고, 나아가 개인이 관리하는 문서·SNS 글·캘린더 등 **사용자의 모든 활동과 기억**을 기록한다. 이렇게 원본으로 저장한 데이터를 시간선상의 공통점과 의미상의 공통점에 따라 추출·가공해 여러 맥락의 단위로 묶고, link로 이어진 tree/graph 구조로 잘 정리해 운영·관리한다. 개인의 지식은 외부 검색이나 자료 추가로 스스로 확장해 간다.

## 2. 4계층 데이터 모델

데이터는 아래로 갈수록 가공·구조화 수준이 높아진다. 상위 계층은 하위 계층을 대체하지 않고 **참조(link)** 한다.

| 계층 | 이름 | 내용 | 관계 |
| --- | --- | --- | --- |
| **L0** | **Record** (raw) | conversation(우선), 문서·SNS·캘린더 등 모든 활동/기억 원본 | 불변, 즉시 저장 |
| **L1** | **Capsule** (캡슐) | 시간적 공통점 + 의미적 공통점으로 묶인 맥락 단위 | Record 1개 → Capsule N개 (many-to-many) |
| **L2** | **Personal Knowledge** | Capsule에서 추출된 node·link의 개인 graph. 사실/의견 분리. 새 정보의 의미적 연관성에 따라 link를 다시 연결·재구성하며 외부 자료로 스스로 확장 | Capsule → node 출처 추적, node ↔ node link |
| **L3** | **Public Knowledge** | 외부 공개 KG에 정렬된 anchor + 다수 개인에게 공통된 핵심 사실 | public anchor → personal node link (메타데이터 보유) |
- **Record**: 가공되지 않은 원본 1건 (출처·시각·소유자 보유)
- **Capsule** (캡슐): 여러 Record를 하나의 맥락으로 묶은 단위 (한 Record는 여러 Capsule에 동시 소속 가능)
- **Knowledge node**: 개인 지식 graph의 한 점 (엔티티·개념·사실·경험)
- **Link**: node 간 또는 계층 간 연결 (관계 타입 + 메타데이터)
- **Anchor**: public knowledge의 node. 외부 공개 KG 식별자(예: Wikidata QID)에 정렬되며, 그 주제를 잘 아는 personal node들로 링크된다

> **Personal Knowledge의 출발점** — Personal Knowledge graph는 0-base(빈 상태)에서 시작한다. 다만 그 구조와 특성은 **public knowledge graph를 참조·활용한다** — 개인 지식은 처음부터 공개 지식의 구조 위에서 자라난다.
> 

> **대화 중 계층 활용** — Agent가 사용자와 대화할 때는 **L0(Record)·L1(Capsule)을 주도적으로** 사용하고, **필요할 때 L2(Personal Knowledge)까지**를 고려 대상으로 삼는다. L3(Public Knowledge)는 "누가 가장 잘 아는지" 라우팅이 필요할 때 닿는다.
>

## 3. 데이터가 흐르는 과정

원본을 받는 경로(빠름)와 가공하는 경로(무거움, 비동기 배치)를 분리한다.

```
sources ─▶ [ingest] ─▶ Record(L0) ──즉시저장──▶ 저장소
                                  │
                  (가공 경로 · 이벤트/주기 트리거)
                                  ▼
   [캡슐화]      시간적·의미적 공통점으로 묶기         ─▶ Capsule(L1)
                                  ▼
   [knowledge]  추출 → node/link, 사실·의견 분리       ─▶ Personal KG(L2)
                                  ▼
   [enrichment] 외부 검색·자료 수집·가공으로 자체 확장  ─▶ L2 확장
                                  ▼
   [public]     외부 KG 정렬 + 공통사실 + 품질 게이팅   ─▶ Public KG(L3)
                                  ▼
   [routing]    "누가 가장 잘 아는지" 자동 라우팅
```

1. **Ingest** — 들어온 데이터를 가공 없이 Record로 즉시 저장한다.
2. **시맨틱 캡슐화 (chunking)** — Record를 시간적·의미적 공통점에 따라 맥락 단위 Capsule로 묶는다. 단순 분할·요약이 아니라, 긴 대화 스트림에서 **시간이 흐르며 사고가 발전한 과정과 인과적 연결**을 살려 맥락을 하나의 캡슐로 담아낸다(내러티브 증류).
3. **Knowledge 추출** — Capsule에서 node·link를 뽑고, **검증 가능한 사실**과 **개인적 의견·경험**을 분리한다.
4. **Enrichment** — 빈 곳·신규 토픽을 외부 검색·자료로 보강해 개인 지식을 스스로 확장한다.
5. **Public 형성** — 4장 절차에 따라 외부 KG 기반 anchor를 구성하고 개인 지식으로 링크한다.
6. **Routing** — 질의가 들어오면 anchor로 토픽을 식별하고, link 메타데이터로 가장 적합한 개인·단체를 찾는다.

> 참고하는 두 가지 관점 — **내러티브 증류**: 최신 내용을 기록·요약하는 데 그치지 않고 사고의 발전과 인과를 서사로 구조화해 압축한다(2단계). **동적 시냅스 재연결**: 생성된 node를 시간 순서·거리에 얽매이지 않고, 새 정보가 들어올 때마다 의미적 연관성에 따라 실시간으로 다시 연결·재구성한다(graph 운영).
>

## 4. Public Knowledge는 어떻게 만들어지는가

Public knowledge는 **밑에서 위로 쌓는 것이 아니라, 외부 공개 KG라는 골격(backbone) 위에 개인 지식을 매달아 잇는 구조**다. 다섯 단계로 나뉜다.

### ① Backbone import — 외부 공개 KG에서 골격을 가져온다

외부 공개 KG에서 핵심 엔티티·상위 클래스를 anchor로 가져온다. **1순위 소스는 Wikidata** — 공개 KG 중 유일하게 CC0(퍼블릭 도메인)이고 다국어이며 100M+ 엔티티와 기계판독 라이선스를 갖췄다. 각 anchor는 외부 식별자(Wikidata QID 등)를 **그대로 보존**해 상호운용성과 다국어를 무료로 확보한다.

| KG | triples | 클래스 | 라이선스 | 본 프로젝트에서의 역할 |
| --- | --- | --- | --- | --- |
| **Wikidata** | ~748M (현재 ~1.4B) | ~300K | **CC0** | **backbone 1순위** (anchor 정렬 기준) |
| DBpedia | ~412M | 736 (수작업) | CC-BY-SA | 클래스 큐레이션 참고 |
| YAGO | ~1.00B | ~570K (자동) | CC-BY 3.0 | 자동 인스턴스화 참고 |
| Freebase | ~3.12B | — | CC-BY 3.0 | 동결·덤프 |
| OpenCyc | ~2.4M | 116,822 | CC-BY 3.0 | 폐기, 희소 스키마 반례 |

### ② 외부 데이터 수집·정제 — 골격에 없는 부분을 채운다

신규 토픽이나 외부 KG에 없는 영역은 외부 검색·자료를 빠르게 수집·가공해 anchor 후보를 만든다. 가능하면 Wikidata 엔티티로 매핑하고, 대응이 없으면 동일 스키마 규칙으로 자체 anchor를 만든다.

### ③ 개인 지식 연결 (자동) — 골격에 개인 지식을 매단다

각 personal knowledge node를 알맞은 public anchor에 연결한다. 이 과정은 **자동화한다.** 연결 link에는 이 개인이 이 주제를 **언제 / 어떻게 / 얼마나** 아는지 기록한다:

- **언제 (temporal)** — 알게 된/갱신된 시점, 최신성
- **어떻게 (provenance)** — 어떤 Record·Capsule·외부 자료에서 나왔는지 출처
- **얼마나 (depth/confidence)** — 지식의 깊이·신뢰도·빈도

이 메타데이터가 곧 **전문가 라우팅의 근거**다.

### ④ 공통 사실 승급 — 공통된 핵심만 공개로 올린다

여러 개인이 같은 anchor에 대해 공유하는 사실 중 **검증 가능한 사실**만 public 핵심 사실 후보가 된다. 개인적 의견·경험은 승급하지 않는다. 승급은 품질 기준(정확성·신뢰성 등)으로 게이팅하고, 개인 출처는 익명 또는 권한 기반 link로만 연결해 프라이버시를 보호한다.

### ⑤ Routing — anchor로 "누가 가장 잘 아는지" 찾는다

질의가 들어오면 public anchor로 토픽을 식별하고, 연결된 personal link들의 메타데이터(언제·어떻게·얼마나)로 후보의 순위를 매긴다. 최신성·깊이·신뢰도가 높은 개인·단체, 또는 새로운 내용을 가장 잘 아는 곳을 찾는다. 결국 public knowledge는 **"기본 사실 + 그 사실을 깊이 아는 곳으로의 anchor"** 로 동작한다.

## 5. 설계 방향

**스키마 — 하이브리드(영역별 분리).** 수작업 큐레이션(DBpedia 736 클래스, 밀집)과 자동생성(OpenCyc 116K 클래스, 희소)의 트레이드오프를 영역으로 나눠 양쪽 장점만 취한다.

- **Backbone** (큐레이션 성격): 상위 클래스·핵심 엔티티는 Wikidata QID 정렬 → 안정적이고 상호운용 가능한 골격
- **확장** (자동생성): 개인 영역·신규/롱테일 사실은 자동 추출로 커버리지 확보
- **승급 게이팅**: personal → public 승급은 품질 기준으로 검증, anchor 연결은 자동화

**저장소 — 단계적 진화.** 데이터 모델은 처음부터 graph link와 외부 KG 정렬을 전제로 설계한다. 규모와 요구가 커지면 저장 방식을 단계적으로 진화시키되, 모델 변경은 최소화한다. (구체적 저장소·도구 선택은 추후 결정)

## 6. 단기 목표

| 단계 | 목표 |
| --- | --- |
| 1 | Conversation 수집·즉시저장 (Record, conversation 우선) |
| 2 | 시맨틱 캡슐화 (chunking) — 시간적·의미적 공통점으로 맥락 단위 Capsule 구성 |
| 3 | Personal Knowledge — node·link 추출, 사실·의견 분리, graph 운영 |
| 4 | Enrichment — 외부 검색·자료로 개인 지식 자체 확장 |
| 5 | Public Knowledge — Wikidata anchor 정렬, 공통사실 승급, 프라이버시 정책 |
| 6 | Expert routing — "누가 가장 잘 아는지" 자동 라우팅 (비전 완성) |
| 중장기 | 소스 확장(문서·SNS·캘린더), 저장 방식 진화 |

## 검토 중인 항목

- **프라이버시 정책 상세** — 사실/의견 분리를 기반으로 하되, 공통사실 승급 시 익명화·권한·동의 범위
- **저장 방식 진화 기준** — 저장 방식을 단계적으로 바꿀 판단 기준
- **품질 기준 선정** — 승급 게이팅에 실제로 사용할 품질 기준

> 방법론·도구(저장소·모델 등 구체 기술) 선택은 비전 확정 후 별도로 다룬다.
